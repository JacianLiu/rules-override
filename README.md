# rules-override

这个仓库用于维护我的 **mihomo / Sub-Store 覆写脚本** 和少量自定义补充规则。

目标是把常用的代理分组、节点分类、DNS 行为、国内外分流规则统一收敛到一套可维护的脚本中，便于在不同客户端里复用。

## 当前推荐入口

### 主维护版本
- `mihomo-override-v2-refactor.js`

这是当前**主要维护和使用**的版本。

相比旧版，它把配置拆成了更清晰的几个层次：
- 静态配置常量
- 节点分类逻辑
- rule-provider / rules 构建逻辑
- 最终装配逻辑

这样做的好处是：
- 更容易看懂每一部分在做什么
- 更适合后续继续调整 DNS、规则集和分组策略
- 改动时更不容易把逻辑搅在一起

### 兼容/对照版本
- `mihomo-override-v2.js`

这个文件仍然保留，主要用于：
- 保留旧版结构
- 作为重构版的对照参考
- 在必要时提供兼容入口

如果后续需要调整核心分流逻辑，**优先修改 `mihomo-override-v2-refactor.js`**。如果旧版仍需继续保留同等行为，再同步修改 `mihomo-override-v2.js`。

## 这套脚本主要做什么

### 1. 节点分类与代理分组
脚本会根据节点名称中的关键字，对节点做基础分类，例如：
- 美国
- 香港
- 新加坡
- 日本
- 家宽节点
- 其他节点
- 中转节点

并生成常用代理组，例如：
- `🚀 节点选择`
- `🤖 AI 服务`
- `✨ Gemini`
- `Ⓜ️ 微软服务`
- `🍎 苹果服务`
- `🔒 国内服务`
- `🐟 漏网之鱼`
- `🛑 广告拦截`

### 2. DNS / Fake-IP / Geodata 配置
脚本内置了较完整的 DNS 配置，包括：
- fake-ip 模式
- 国内/代理分流的 nameserver 策略
- geodata 自动更新
- `geoip.dat` / `geosite.dat` / `country.mmdb` / ASN 数据地址

同时会根据客户端场景，对 DNS 行为做一定区分。

### 3. 国内规则简化
当前国内规则已经尽量改为直接使用 geodata：
- `GEOSITE,geolocation-cn`
- `GEOSITE,cn`
- `GEOIP,CN`

这意味着一部分原先依赖 rule-provider 的国内规则，已经被 dat 直读方式替代，维护成本更低。

### 4. 本地补充规则
并不是所有国内规则都适合完全交给 `geosite.dat` / `geoip.dat`。

目前仍保留：
- `rules/list/supplement-cn.list`

它用于放置 **GEOSITE/CN 不能完整覆盖**，但又希望继续归入“国内服务”的补充域名规则。

## 使用方式

这个仓库主要给 **Sub-Store 覆写脚本** 使用。

默认建议：
- 优先使用 `mihomo-override-v2-refactor.js`

典型方式是：
- 在 Sub-Store 中将该脚本作为覆写脚本引用
- 让客户端在订阅处理阶段自动生成最终配置

如果你只是想复用当前策略，通常只需要引用重构版脚本即可。

## 当前规则策略说明

### 国内规则
当前国内流量的主要思路是：
- 直接使用 `GEOSITE,geolocation-cn`
- 直接使用 `GEOSITE,cn`
- 直接使用 `GEOIP,CN`
- 额外用 `supplement-cn.list` 做少量补充

这样可以减少对单独 `.mrs` 国内规则文件的依赖。

### 补充规则文件格式
当前自维护国内补充规则使用：
- 文本 list：`rules/list/supplement-cn.list`

不再继续维护对应的 `.mrs` 版本。

## 维护约定

### 1. 优先修改重构版
涉及以下内容时，优先修改：
- 代理组结构
- 节点分类逻辑
- DNS 策略
- rule-provider 定义
- 国内/国外分流规则

目标文件是：
- `mihomo-override-v2-refactor.js`

### 2. 旧版按需同步
如果某次修改会影响实际行为，而旧版脚本仍然要继续保留相同策略，那么需要同步更新：
- `mihomo-override-v2.js`

### 3. 临时分析文件不要入库
规则覆盖分析、临时导出、实验脚本等内容，默认不应长期保留在仓库中。仓库应尽量只保留：
- 可直接使用的覆写脚本
- 正在维护的规则文件
- 必要的说明文档

### 4. 规则调整以实际覆盖结果为准
像 `GEOSITE` / `GEOIP` 替换这类调整，应先验证覆盖情况，再决定是否删掉原有补充规则，而不是直接凭感觉替换。

## 注意事项

- 这是一个偏个人使用场景的规则仓库，部分策略明显带有定制属性。
- 某些 DNS、分流、直连/代理选择，不一定适合所有网络环境。
- WebRTC、TCP/IP 指纹、系统指纹一致性这类问题，很多时候取决于：
  - 出口节点本身
  - 浏览器行为
  - 客户端环境
  而不只是规则写法。
